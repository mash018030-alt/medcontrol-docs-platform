# Устройство репозитория: движок vs контент

Краткая шпаргалка, чтобы не путать **код платформы**, **регламенты про движок** и **тексты статей MedControl**.

| Путь | Что это |
|------|---------|
| **`docs-app/`** | Веб-приложение (React + Vite): UI, роутинг, поиск, сборка. Это **движок сайта документации**, а не папка со статьями для читателя. |
| **`docs-app/public/content/`** | Пользовательский контент в этом репозитории: `.md` статей (`0_docs/…`), картинки, `references/` с PDF-эталонами и т.д. Раньше — отдельный репозиторий на GitHub. |
| **`docs-app/engine-docs/`** | Служебные **markdown-заметки только про движок** (PDF, таблицы, деплой). **Не** попадают на сайт и **не** в поиск по статьям. Имя папки намеренно не `docs`, чтобы не смешивать с пользовательскими «доками» в `public/content/`. |
| **`docs-brand/`** | Референс бренда платформы доков (палитра и т.п.) в корне репо — не про статьи и не «дизайн продукта MedControl». |

Индекс файлов в `engine-docs/`: **[`docs-app/engine-docs/README.md`](docs-app/engine-docs/README.md)**.

---

## Опционально: переименовать каталог `docs-app`

Исторически каталог приложения называется **`docs-app`**. Если нужно явнее отделить его от пользовательских «доков» в `public/content/`, можно переименовать, например, в **`doc-platform`**:

1. Закройте IDE/терминалы с текущей рабочей папкой внутри `docs-app` (иначе Windows может вернуть «папка занята»).
2. В корне репозитория:
   - `git mv docs-app doc-platform`  
     (или переименование в Проводнике, затем `git add -A`).
3. Если снова понадобится **git submodule** для контента — заведите **`.gitmodules`** и путь вида `doc-platform/public/content` (сейчас контент в репо без submodule).
4. Массово замените в репозитории строку **`docs-app`** на **`doc-platform`** (CI: `.github/workflows/*.yml`, **`.github/dependabot.yml`**, **`.vscode/tasks.json`**, **README.md**, **CONTEXT.md**, **brief.md**, комментарии в скриптах, **`.cursor/rules/*.mdc`**, тексты в **`doc-platform/engine-docs/`**).

Пока переименование не сделано, везде в инструкциях остаётся каноническое имя **`docs-app`**.

## Другие имена, которые иногда путают

| Имя | Почему может сбивать | Переименовывать? |
|-----|----------------------|------------------|
| **`public/content/0_docs/`** | Слово **docs** рядом с «документацией про движок»; префикс **`0_`** выглядит как служебная нумерация. По смыслу это **дерево разделов пользовательской документации MedControl** (URL, `nav.js`). | **Почти никогда:** смена затронет все пути статей, меню, редиректы, внешние закладки. Имеет смысл только как большая миграция с ТЗ. |
| **`public/content/references/`** | «References» ассоциируется со списком литературы; здесь **эталоны, PDF, экспорты руководств** для сверки. | Переименование = массовая правка путей в контенте. Проще опираться на **`references/README.md`**. |
| **`docs-brand/`** (корень репо) | Явное имя: **бренд / палитра UI платформы документации**, не пользовательский контент. | См. **`docs-brand/README.md`**. |
| **`public/`** (стандарт Vite) | Статика, в т.ч. **`public/content/`**. | Не трогать корень `public/` без нужды: конвенция фреймворка. |
| **`archive/`** | Обычно однозначно: **история миграций**, не актуальный код. | Можно оставить; при желании — `migration-archive/` (косметика). |
| **`brief.md` / `CONTEXT.md`** (файлы, не папки) | Оба в корне; новичок может не знать порядок чтения. | Не переименовывать без нужды; в **`README.md`** уже есть навигация по ролям. |

Итого: после выделения **`engine-docs/`** самые «опасные» по смыслу остаются **`0_docs`** и **`references`** внутри контента — их трогают только при осознанной миграции путей, а не «для красоты».
