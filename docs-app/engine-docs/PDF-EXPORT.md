# Экспорт в PDF: правила и настройки

Единая точка правды по текстовому PDF (Playwright) и запасным путям. **Перед правками PDF читайте этот файл** и `pdf-server/README.md`.

**Агент Cursor:** при любых изменениях кода или конфигурации, влияющих на PDF, сначала сверяться с этим документом; если поведение или инварианты меняются — **обновить этот файл** и при необходимости `pdf-server/README.md`, `engine-docs/SECTION_PDF_BUNDLE.md` (см. правило `.cursor/rules/docs-pdf-playwright.mdc`).

---

## 1. Архитектура (не менять смысл)

| Слой | Роль |
|------|------|
| **Канон** | Headless Chromium печатает **ту же страницу**, что видит пользователь, с `?mc_pdf=1` (упрощённая вёрстка: без шапки/сайдбара). |
| **Сервис** | `docs-app/pdf-server` — `POST /api/pdf`, тело `{ "url": "<полный https? URL страницы с mc_pdf=1>" }`, ответ — бинарный PDF. |
| **Запасной путь** | `html2pdf` / html2canvas в браузере — **растровый** PDF, текст плохо выделяется. Только если сервис недоступен или узкий экран (см. ниже). |

---

## 2. Локальная разработка (инварианты)

1. **Same-origin в DEV.** Браузер вызывает **`POST /api/pdf`** на **том же origin**, что и Vite. Прокси в `vite.config.js` пересылает на `http://127.0.0.1:3001`.
2. **Не направлять браузер** с `http://localhost:5174` (или 5175…) **напрямую** на `http://127.0.0.1:3001` — другой origin → часто `Failed to fetch` (CORS / Private Network Access). В `.env.local` **не нужен** `VITE_PDF_SERVICE_URL=http://127.0.0.1:3001` для локалки.
3. Логика URL API: **`getPdfServiceApiUrl()`** в `src/utils/runArticlePdfExport.js` — в `import.meta.env.DEV` при пустой или «локальной» базе возвращает **`/api/pdf`**.
4. Запуск: из `docs-app` — **`npm run dev:with-pdf`** (рекомендуется) или два процесса: `npm run pdf-server` и `npm run dev`.
5. Проверка сервиса: **`http://127.0.0.1:3001/health`** → `{"ok":true}`.
6. **Прокси Vite:** при ошибке соединения с pdf-server (например `ECONNREFUSED`) обработчик в `vite.config.js` отдаёт **JSON** с полем `error`, а не HTML «Internal Server Error» — иначе фронт не покажет причину.
7. **Разбор ошибки fetch:** в `runArticlePdfExportPlaywright` тело ответа читается **один раз** через `res.text()`, затем при необходимости `JSON.parse` — после неудачного `res.json()` второй вызов `text()` часто пустой.

---

## 3. Production / статический хостинг

- Прокси Vite **нет**. Нужен публичный **`VITE_PDF_SERVICE_URL`** на этапе **`npm run build`** (без завершающего `/`).
- На pdf-server: **`ALLOWED_HOSTS`** (хост:порт сайта документации), **`CORS_ORIGIN`** — точный Origin фронта.
- Подробности и шаблоны: **`pdf-server/README.md`**, **`docs-app/.env.example`**.

## 4. Где какой PDF

| Сценарий | Механизм | Ключевые файлы |
|----------|----------|----------------|
| Статья документации | `runArticlePdfExport` → URL текущей страницы + `mc_pdf=1` | `Article.jsx`, `runArticlePdfExport.js` |
| Статья новостей | Тот же `runArticlePdfExport` | `NewsArticle.jsx` |
| **PDF всего раздела** (кнопка на разводящей / карточке дашборда) | Отдельный маршрут **`/section-pdf-bundle?root=…&mc_pdf=1`**, Playwright печатает его | `SectionPdfBundlePage.jsx`, `runArticlePdfExport.js` → `buildSectionBundlePrintUrl`, `SectionCard.jsx` |
| Раздел: состав markdown | Порядок как «Предыдущая / Следующая», **без** разводящей **`…/00_main`** | `orderedPathsForSectionPdfBundle` в `src/data/nav.js` |
| Узкий экран (≤1023px) | `preferClientSideArticlePdf()` — для **одной статьи** сразу html2pdf; для **сборки раздела** — открытие вкладки печати | `runArticlePdfExport.js` |

Новости: статические PDF из `news-tree.json` **не используются**; иконка ведёт на страницу релиза и кнопку «Скачать в PDF».

### Иллюстрации в статьях (без каруселей)

В контенте **нет каруселей**: в статье идут обычные **отдельные картинки** (`<img>`). На это опираются оба канала PDF:

| Канал | Поведение |
|-------|-----------|
| **Playwright** | Печатается текущая вёрстка; скрытых слайдов и переключателей карусели нет — ничего дополнительно не раскрывать. |
| **html2pdf** (запасной) | В `pagebreak.avoid` учитываются типичные узлы (`img`, заголовки, списки, таблицы, коллауты, сноски). Под карусели отдельных правил **нет** и не требуется. |

Если когда-нибудь **вернуть** карусели в markdown/HTML, нужно будет отдельно продумать: видимый в PDF набор слайдов, `pagebreak` для html2pdf и при необходимости подготовку DOM перед снимком (как раньше делал клиентский скрипт к карусели).

## 5. pdf-server (серверная часть)

- В запросе — полный URL; хост страницы должен быть в **`ALLOWED_HOSTS`** (по умолчанию расширенный список портов Vite/preview в `server.mjs`).
- Ожидается узел **`.docs-markdown-root`** на странице (таймаут ожидания — см. код).
- **Одна длинная страница:** пороги `PDF_SINGLE_PAGE_MAX_PX` и **`PDF_MAX_SINGLE_PAGE_HEIGHT_MM`** (лимит Chromium ~5080 mm; иначе 500 и падение — см. комментарии в `server.mjs`).
- Для навигации: `localhost` / `::1` в URL могут приводиться к **`127.0.0.1`**, чтобы headless стабильно достучался до Vite на Windows.

Детали переменных: **`pdf-server/README.md`**.

## 7. Что нельзя делать без явного ТЗ

- Отключать запасной html2pdf без замены UX.
- Менять контракт `POST /api/pdf` без синхронного обновления фронта и сервера.
- В DEV снова включать прямой вызов `:3001` из браузера вместо `/api/pdf`.
