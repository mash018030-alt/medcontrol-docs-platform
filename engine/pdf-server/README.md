# PDF-сервис (Playwright)

Генерирует PDF той же вёрсткой, что в Chromium (печать страницы), в отличие от html2canvas в браузере.

Общая архитектура фронта + сценарии (статьи, новости, сборка раздела): **[`../engine-docs/PDF-EXPORT.md`](../engine-docs/PDF-EXPORT.md)**.

## Запуск локально

1. Установка (из каталога `pdf-server`):

```bash
npm install
```

2. Поднять сервис:

```bash
npm start
```

По умолчанию порт **3001**. Разрешённые хосты страниц по умолчанию: типичные порты Vite **5173–5200** для `localhost` и `127.0.0.1`, плюс `4173–4175` (часто `vite preview`), `3000` — чтобы при смене порта фронта Playwright не получал 403.

3. Запустите фронт в **`engine`** одним из способов:

- **Рекомендуется:** `npm run dev:with-pdf` — поднимает **и** pdf-server, **и** Vite (один терминал).
- Или два терминала: `npm start` в `pdf-server` и `npm run dev` в `engine`.

Порт фронта по умолчанию **5174** (`vite.config.js`). Vite **проксирует** `POST /api/pdf` → `http://127.0.0.1:3001`, запрос из браузера идёт на **тот же origin**, без cross-origin на `:3001`.

4. **`VITE_PDF_SERVICE_URL` в `.env.local` для локального `http://127.0.0.1:3001` не нужен** и раньше мешал: браузер ходил с `localhost:5174` на `127.0.0.1:3001` → часто **TypeError: Failed to fetch**. В режиме `npm run dev` движок всё равно использует **`/api/pdf`** (прокси), если переменная пустая или указывает на localhost/127.0.0.1.

Задавайте `VITE_PDF_SERVICE_URL` в DEV только для **удалённого** сервиса печати (полный `https://…`).

5. Проверка: откройте `http://127.0.0.1:3001/health` — ответ `{"ok":true}`. Если кнопка PDF даёт «Failed to fetch» — не запущен `pdf-server`, занят порт **3001**, или фронт открыт не с того хоста/порта (см. `ALLOWED_HOSTS` в логе при старте сервера).

Если `pdf-server` недоступен, для **одной статьи** скачивается **растровый** PDF (html2pdf); качество переносов хуже, чем у Chromium.

## Продакшен

### GitHub Pages (и любой чисто статический фронт)

Сборка в `dist/` **не включает** `pdf-server`: на страницах вида `*.github.io` прокси `/api/pdf`, как в `npm run dev`, **нет**. Без переменной **`VITE_PDF_SERVICE_URL`** при `npm run build` фронт не знает, куда слать `POST`, появится предупреждение, а кнопка «PDF» у статьи уйдёт в **растровый** запасной вариант (html2pdf).

1. Задеплойте `pdf-server` на отдельный хост с HTTPS.
2. В настройках сервиса укажите **`ALLOWED_HOSTS`** — хост вашей документации (например `mash018030-alt.github.io`; если сайт в подпапке репозитория, хост всё равно тот же, важен полный URL страницы).
3. **`CORS_ORIGIN`** — точный Origin фронта, например `https://mash018030-alt.github.io` (без слэша в конце).
4. Перед сборкой фронта: `VITE_PDF_SERVICE_URL=https://ваш-сервис...` (без завершающего `/`), затем `npm run build` и публикация `dist`.

Шаблон комментариев см. в `engine/.env.example`.

- Разверните `pdf-server` на том же VPC/хосте или рядом (Node + зависимости Playwright тяжелее обычного API).
- Сервис открывает в Chromium **тот же URL**, что у пользователя, с параметром `mc_pdf=1` (без шапки и бокового меню). Хост страницы и хост API могут различаться; главное, чтобы с **машины, где крутится pdf-server**, этот URL был **доступен по сети** (localhost только если фронт и API на одной машине).
- Задайте `ALLOWED_HOSTS` списком хостов **вашего** сайта документации (через запятую), чтобы исключить SSRF.
- Задайте `CORS_ORIGIN` на точный Origin фронта, не `*`, если возможно.
- В прод-сборке фронта укажите `VITE_PDF_SERVICE_URL` на публичный URL сервиса (или прокси с того же домена, например `/api/pdf`).

## Переменные окружения

| Переменная                  | Описание |
|-----------------------------|----------|
| `PORT`                      | Порт сервера (по умолчанию 3001) |
| `ALLOWED_HOSTS`             | Разрешённые `host:port` для поля `url` в запросе |
| `CORS_ORIGIN`               | Заголовок `Access-Control-Allow-Origin` |
| `PDF_SINGLE_PAGE_MAX_PX`         | Порог высоты `.docs-markdown-root` в px для попытки **одной длинной страницы** (по умолчанию `24000`). Если расчётная высота в мм превышает лимит Chromium, сервис сам уйдёт в многостраничный A4. |
| `PDF_MAX_SINGLE_PAGE_HEIGHT_MM`  | Максимальная высота одной «листовки» в мм (по умолчанию `5000`, не больше ~5080 — иначе Chromium падает с 500). Длинные релиз-ноуты и статьи уходят в многостраничный A4. |
